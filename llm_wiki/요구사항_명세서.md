## 1. 문서 개요

### 1.1 목적
법인카드 정산 업무(입력·검토·확정·정책결정)를 3개의 AI Agent(초안 작성 / Rule / Risk Review)와 사람의 최종 확정으로 자동화하는 시스템의 기능적/비기능적 요구사항을 정의한다.

### 1.2 범위
- **포함(MVP)**: 정산 초안 자동생성, **가맹점 업종 자동 판별(비용분류 보조)**, Rule 기반 1차 자동판정, **단순 이상거래 탐지(1차) + RAG 내규 기반 검증(2차)** 위험 검토, 역할별 대시보드, ERP **전표(안)** 생성.
- **제외(MVP)**: ERP 직접 적재/연동, 별도 메시지 큐(Redis/Celery), 별도 OCR 엔진, 멀티테넌시(Open Issue로 별도 확정).

### 1.3 용어 정의
| 용어 | 정의 |
|---|---|
| Draft Agent | 거래·영수증을 받아 정산 초안(분류·증빙매칭·규정힌트)을 자동 완성하는 Agent |
| Rule Agent | 규정을 실행 가능한 Rule로 생성→검증→적용하는 Agent |
| 룰 그래프(RuleGraph) | 단건 룰(노드)들이 트리/그래프로 조립된 **최종 상태 도메인**. ACTIVE·버전관리·시뮬레이션·롤백의 단위 |
| 룰(노드) | 단건 룰. `condition + action + next_routings`로 구성되는 그래프의 노드 |
| next_routings | 각 룰 노드가 평가 결과(MATCH/NO_MATCH/PASS/REJECT/REVIEW)에 따라 다음에 평가할 노드로 향하는 라우팅(에지) 구조 |
| Risk Review Agent | Rule로 걸러지지 않은 미매칭/불확실 건을 **① 단순 이상거래 탐지로 1차 선별**하고, **② 이상 후보에 대해 RAG로 내규·세법 근거를 검색해 위반 여부를 검증**한 뒤 검토 우선순위를 부여하는 Agent |
| SoR (System of Record) | 확정 데이터의 단일 진실 원천. 본 시스템에서는 Django가 관리하는 PostgreSQL |
| RPA 1차판정 | Rule Engine에 의한 결정론적 1차 자동 판정 단계 |
| RAG | Retrieval-Augmented Generation. 규정/사례 문서를 벡터 검색해 LLM 근거로 사용 |
| 가맹점 업종 구분 시스템 | 가맹점명으로 업종(업태)을 판별하는 서브시스템. 자체 DB 캐시 → 카카오 지도 API → 웹검색 캐스케이드로 확정하며, 비용분류 제안의 보조 입력으로 쓴다(세무 기준 아님) |
| MCC (Merchant Category Code) | 카드사가 부여하는 표준 가맹점 업종코드. 세무·회계 기준에 부합하나 MVP 범위 밖(카드사 제휴 필요) — post-MVP 확장 |
| 팀(제출 단위) | 조직도상 팀이 아니라 정산을 취합해 올리는 단위(1인도 가능) |

## 2. 시스템 개요

### 2.1 문제 정의
법인카드 정산은 모든 조직이 매달 반복하지만 누구도 본업으로 여기지 않는 저부가 업무이며, 임직원(입력 번거로움) → 팀장(취합 부담) → 회계 담당자(전건 수작업 검토) → 상부(감으로 정책 결정)까지 전 역할의 시간을 소모시킨다.

핵심 문제 3가지: ① 입력이 번거롭다, ② 검토가 전건 수작업이다, ③ 의사결정 근거가 없다.

### 2.2 해결 방향
① 입력을 AI가 대신하고, ② 확실한 건은 규칙으로 자동 처리하며, ③ 애매한 건만 근거와 함께 골라주고, ④ 결정 근거를 대시보드로 제공한다. **예산·정책은 통제(차단)가 아니라 지표·추천으로만 반영한다.**

### 2.3 설계 원칙 (아키텍처 관점)
- 관심사 분리: 트랜잭션/도메인(Django)과 AI 오케스트레이션(FastAPI)을 물리적으로 분리
- SoR은 Postgres 하나: AI는 "제안"만 생성하고, 확정은 Django 도메인을 통해서만 이뤄짐
- Agent는 LLM + Tools(MCP): 모든 Agent는 단일 FastMCP 서버를 통해서만 데이터 접근 (LLM 직접 DB 접근 금지)
- MVP는 동기 REST: 메시지 큐 없이 동기 호출, 무거운 작업(임베딩/학습)은 관리자 트리거 배치
- **사람 확정 원칙**: 자동 통과 건도 최종 확정은 사람이 수행. 시스템은 상태만 "승인대기"로 전이

### 2.4 3-Agent 개요
| Agent | 대상 | 역할 |
|---|---|---|
| ① 초안 작성 Agent | 임직원/팀 | 지출 입력 단계의 시간을 줄인다 |
| ② Rule Agent | 회계 담당자 | 확신 가능한 건의 자동 판정 엔진 |
| ③ Risk Review Agent | 회계 담당자 | 사람 검토를 보조(이상탐지 1차 선별 → RAG 내규 검증 → 우선순위 + 근거) |

## 3. 이해관계자 및 사용자 역할

| 역할 | 설명 | 주 사용 화면 |
|---|---|---|
| 사용자(임직원) | 실제 카드를 사용하고 지출을 발생시키는 개인 | 내 지출 / 증빙 제출 |
| 팀(제출 단위) | 지출을 취합해서 올리는 단위(다인 또는 1인) | 팀 취합·제출 |
| 회계 담당자 | 제출된 정산을 검토·승인·반려·최종 확정하는 실무자 | 검토 워크스페이스, Rule 콘솔 |
| 회계/운영 상부(관리자) | 규정·예산 현황을 보고 정책을 결정하는 의사결정자 | 거버넌스 대시보드 |

> **참고(Open Issue)**: 위 4개 역할 중 "회계 담당자"와 "회계/운영 상부(관리자)"의 시스템 권한 구분(예: Rule 승인 권한, 규정 문서 업로드 권한 등 세부 RBAC 경계)은 원본 두 문서에 명확히 분리되어 있지 않다. 후속 검토에서 역할별 권한 매트릭스로 구체화가 필요하다.

## 4. 핵심 도메인 객체 요구사항

### 4.1 카드 구분 (Card Type)
| 카드 구분 | 정의 | 정산 특성 |
|---|---|---|
| 개인 배정 카드 | 특정 임직원에게 배정된 법인카드 | 사용자=정산 책임자 명확 |
| 팀 카드(팀 공용) | 팀 단위로 공유 | 사용 건별 실사용자·목적 지정 필요 |
| 공용 카드 | 부서/법인 공용, 다수 사용 | 사용자 귀속 모호 → AI 사용자 추정·확인 필요 |
| 후정산 카드/지출 | 개인 선지출 후 사후 청구 | 영수증 기반, 증빙 의존도 높음 |
| 선결제/충전형(선불) | 사전 충전·한도형 결제 수단 | 잔액·충전 이력 관리 |

**요구사항**: 카드 구분에 따라 사용자 귀속 방식과 증빙 요구 수준이 자동으로 달라져야 한다(예: 공용 카드는 실사용자 확인 단계 자동 삽입, 후정산은 증빙 필수 검증 자동 삽입).

### 4.2 비용 분류 (Category)
| 분류 | 예시 | 대표 규정/증빙 포인트 |
|---|---|---|
| 업무활성(업무추진) | 업무 관련 일반 지출 | 목적·대상 명시 |
| 회의 | 사내/외부 회의, 다과 | 참석자·건당 한도 |
| 식대 | 야근·업무 식사 | 인원수·시간대 |
| 출장 | 교통·숙박·일비 | 출장 신청/구간·일정 |
| 접대 | 거래처 접대·경조 | 거래처·접대 한도·세무 리스크 |
| 비품 | 소모품·사무용품 | 자산화 여부·구매 목적 |
| (확장) | 교육/도서, 복리후생 등 | 분류별 규정 확장 가능해야 함 |

### 4.3 항목별 3요소
각 비용 분류는 **규정(Rule 소스: 한도·조건·필요증빙·세무기준)**, **예산(지표 소스: 배정·소진, 통제 아님)**, **증빙(검증 대상: 영수증/명세/사유서)** 3요소를 가져야 하며, 이 3요소가 Agent 판단의 입력 재료가 된다.

### 4.4 정산 건 라이프사이클 (상태 머신)
```
DRAFT → SUBMITTED → RPA_JUDGED
  ├─(자동통과)──► PENDING_CONFIRM ─(사람 확정)─► CONFIRMED → ERP_VOUCHER_DRAFTED
  ├─(자동반려)──► RETURNED ─(재제출)─► SUBMITTED
  └─(불확실/미매칭)► IN_REVIEW ─► APPROVED / RETURNED / CORRECTED
```
- 모든 상태 전이는 Django 서비스 레이어에서만 수행되고, `settlement_events` + `audit_logs`에 기록되어야 한다.
- Rule이 "확신 통과"로 판정해도 시스템이 자동으로 확정 처리해서는 안 되며, 반드시 회계 담당자의 사람 확정 단계를 거쳐야 한다.

## 5. 기능 요구사항 (FR)

### 5.1 초안 작성 Agent (FR-DA)
| ID | 요구사항 |
|---|---|
| FR-DA-01 | 카드 사용 내역(거래)과 업로드된 영수증 이미지를 입력받아 정산 초안을 자동 완성해야 한다. |
| FR-DA-02 | 영수증 이미지에서 가맹점/금액/일시/품목 필드를 판독하고 거래 건과 자동 매칭해야 한다(별도 OCR 엔진 없이 비전 모델 직접 판독). |
| FR-DA-03 | 가맹점·금액·시간·패턴을 근거로 비용 분류(식대/회의/출장/접대/비품 등)를 자동 제안해야 한다. |
| FR-DA-03a | 비용 분류 제안 시 **가맹점 업종 구분 시스템**이 판별한 업종(업태)을 보조 입력으로 활용해야 한다(§6.5). |
| FR-DA-03b | 가맹점 업종 판별은 **자체 DB 캐시 → 카카오 지도 API → 웹검색** 순의 캐스케이드로 수행하고, 최종 결과를 자체 DB에 캐시(재사용)해야 한다. |
| FR-DA-03c | 가맹점 업종은 **비용분류 보조 힌트**로만 사용해야 하며, 세무·회계 계정과목 판단의 근거로 삼아서는 안 된다(세무 판단은 규정/세법 RAG에 위임). 표준 업종코드(MCC)는 post-MVP 확장(§9 Open Issue). |
| FR-DA-04 | 카드 구분(개인/팀/공용/후정산)에 따라 필요한 추가 입력(실사용자·목적·증빙)을 자동 요청해야 한다. |
| FR-DA-05 | 누락된 증빙을 사용자에게 명시적으로 안내해야 한다. |
| FR-DA-06 | 분류별 규정(한도·필요 서류)을 입력 시점에 미리 안내하여 반려를 예방해야 한다(규정 힌트). |
| FR-DA-07 | 팀원 지출을 한 화면에 모아 누락·이상 건만 강조하는 팀 취합 뷰를 제공해야 하며, 정상 건은 접어(collapse) 표시해야 한다. |
| FR-DA-08 | 팀장은 이상 건만 개별 처리하고 나머지는 일괄 제출할 수 있어야 한다. |
| FR-DA-09 | 초안 결과는 확정 데이터가 아닌 `settlements(DRAFT)` 상태로 저장되어야 하며, 사용자가 보정할 수 있어야 한다. |

### 5.2 Rule Agent — 생성 (FR-RB)
| ID | 요구사항 |
|---|---|
| FR-RB-01 | 관리자가 업로드한 규정 문서를 청킹·임베딩하여 벡터 저장소(Chroma)에 적재해야 한다. |
| FR-RB-02 | RAG 검색으로 규칙화 가능한 조항을 추출해야 한다. |
| FR-RB-03 | 추출된 조항을 바탕으로 실행 가능한 단건 룰(노드) 초안(condition + action)을 LLM이 자동 생성해야 한다. |
| FR-RB-04 | 생성된 단건 룰은 **룰 그래프(트리)의 노드**로 조립되어야 하며, 각 노드는 평가 결과별 다음 노드 라우팅(**next_routings**)을 가져야 한다. |
| FR-RB-05 | 상태·버전 단위는 개별 노드가 아니라 **룰 그래프(최종 상태 도메인)** 여야 하며, 조립된 그래프는 `status=DRAFT`로 저장되어야 한다. |

### 5.3 Rule Agent — 검증 (FR-RV)
| ID | 요구사항 |
|---|---|
| FR-RV-01 | 생성된 룰 그래프 후보는 적용 전 반드시 과거 거래 데이터에 **그래프 단위로** 시뮬레이션되어야 한다. |
| FR-RV-02 | 시뮬레이션 결과로 매칭 건수, 오탐율, 예상 검토 감소량 및 노드별 커버리지 지표를 산출해야 한다. |
| FR-RV-03 | 시뮬레이션 지표는 관리자 검토·승인 화면(Rule 콘솔)에 표시되어야 한다. |
| FR-RV-04 | 관리자가 승인한 **룰 그래프 버전만** `ACTIVE`로 전이될 수 있다(자동 승인 금지). 동시에 ACTIVE인 그래프 버전은 스코프당 하나여야 한다. |
| FR-RV-05 | ACTIVE·버전관리·**롤백은 룰 그래프(최종 상태 도메인) 단위**로 이뤄져야 하며, 관리자는 이전 승인 버전으로 즉시 복원(롤백)할 수 있어야 한다(직전 ACTIVE는 ARCHIVED). |

## 6. AI/ML 및 RAG 요구사항

### 6.1 위험 검토 파이프라인 (MVP: 이상탐지 1차 + RAG 내규 검증 2차)

MVP의 Risk Review는 라벨이 충분히 축적되기 전(콜드스타트) 단계이므로, **지도학습에 의존하지 않는 2단계 구조**로 구성한다.

**1차 — 단순 이상거래 탐지 (비지도)**
- **학습**: `transactions` → feature 생성 → 비지도 이상탐지 모델(예: Isolation Forest 등) 학습 → 모델 레지스트리(버전관리) → 서빙 배포. *(라벨 불필요)*
- **추론**: 실시간(동기) — `get_tx_features` → `ml_infer` → anomaly_score·피처 기여 산출 → 컷오프 이상 건만 "이상 후보"로 선별
- **Feature 예시**: 금액/정규화금액, 시간대/요일/심야여부, 가맹점/업종, 사용자·부서, 최근 N일 빈도, 동일거래처 누적, Rule 적용결과, 증빙 유무

**2차 — RAG 내규 기반 검증 (이상 후보 한정)**
- 1차에서 선별된 이상 후보 건에 대해서만 §6.2 RAG로 사내 규정·세법 근거 및 유사 사례를 검색
- LLM이 거래를 검색된 내규와 대조하여 위반 여부를 검증하고, 검토사유·권장의견을 출처(문서·조항)와 함께 생성 → `risk_reviews` 저장

**피드백(라벨 적재만)**: 회계 담당자 결정 → `decision_labels` 축적. **MVP에서는 자동 재학습을 수행하지 않으며**, 향후 지도학습(review_probability) 도입을 위한 학습 데이터로만 적재한다.

**평가지표**: 이상탐지 검토 적중률·이상탐지 AUC, RAG 근거 정확도, 검토량 감소율 *(목표치는 §9 Open Issue)*

> **확장(post-MVP)**: 라벨이 충분히 쌓이면 지도학습 모델(검토필요확률, 예: XGBoost/LogReg 등)과 `decision_labels` 기반 온디맨드 재학습 루프를 추가해 우선순위 산출을 고도화한다.

### 6.2 RAG 파이프라인
- 문서 업로드 → 청킹 → 임베딩 → Chroma upsert(메타데이터 포함) → 질의 시 쿼리 임베딩 → 유사도 검색(top-k, 메타 필터) → LLM 컨텍스트 주입
- 버전/유효일 메타데이터로 최신 규정을 우선해야 하며, 규정 개정 시 재임베딩(비동기)되어야 한다.
- 근거 표시 시 출처(문서·조항)를 함께 반환하여 회계 담당자가 검증할 수 있어야 한다.

### 6.3 FastMCP Tool 명세
| Tool | 인자 | 출력 | 데이터 원천 | 사용 Agent |
|---|---|---|---|---|
| `get_policy` | category | limit, required_evidence, refs | Postgres(Django 경유) | Draft, Rule, Risk |
| `get_card_context` | card_id | card_type, required_inputs | Postgres(Django 경유) | Draft |
| `classify_merchant` | merchant, place_hint? | industry_code, industry_label, confidence, source | 캐시=Postgres(Django 경유), 미스 시 카카오 지도/웹검색 | Draft, Risk |
| `search_policy` | query, filters | chunks[] | Chroma 직접 | Rule, Risk |
| `search_cases` | tx / query | similar_cases[] | Chroma 직접 | Risk |
| `fetch_historical_tx` | period, filters | tx[] | Postgres(Django 경유) | Rule(검증) |
| `run_rule_engine` | tx, ruleset | decision, confidence, hits[] | 내부 Rule 엔진 | Rule(적용) |
| `get_tx_features` | tx_id | feature_vector | Postgres(Django 경유) | Risk |
| `ml_infer` | feature_vector | anomaly_score, contribs *(MVP: 비지도 이상탐지만. review_prob는 post-MVP)* | ML 모델(FastAPI 내) | Risk |

**요구사항**: 단일 FastMCP 서버가 3개 Agent의 도구를 모두 노출해야 하며, 관계형 데이터는 반드시 Django 내부 API를 경유하고, 벡터 데이터(Chroma)만 직접 접근이 허용된다. LLM/Tool은 Postgres에 직접 SQL을 실행할 수 없다.

### 6.4 LLM/비전 모델 사용 원칙
- 영수증 판독은 별도 OCR 엔진 없이 비전 모델이 이미지에서 직접 필드를 추출한다.
- Rule 적용(RPA) 단계는 결정론적 엔진만 사용하고 LLM은 Rule 생성 단계에서만 사용해 재현성을 확보한다.

### 6.5 가맹점 업종 구분 파이프라인 (Merchant Classification)
가맹점명만으로는 비용분류가 모호한 경우가 많아, 가맹점의 **업종(업태)** 을 별도로 판별해 비용분류 제안(FR-DA-03)과 이상탐지 feature('가맹점/업종')를 보강한다. 외부 조회 비용·지연을 줄이기 위해 **자체 DB 캐시를 1차 소스**로 두는 캐스케이드 구조다.

- **파이프라인**: ① 가맹점명 정규화 → ② 자체 DB 캐시 조회(히트·유효 시 반환) → ③ 카카오 지도 API 장소검색(업종 매핑) → ④ 웹검색 tool(①~③ 실패/모호 시 LLM이 스니펫 해석) → ⑤ 최종 업종 확정 후 자체 DB에 캐시(upsert).
- **출력**: `{ industry_code, industry_label, confidence, source(CACHE/KAKAO/WEB) }`.
- **데이터**: 캐시 테이블 `merchant_categories`(관계형 → Django 소유). 외부 API 호출은 FastAPI(ai)에서 수행하고 결과만 Django 경유로 적재(§6.3 데이터 접근 원칙 준수).
- **한계**: 카카오·웹 기반 업종은 장소·마케팅 분류로 **세무·회계 기준(MCC)과 불일치**할 수 있어, **비용분류 보조 힌트로만** 사용한다. 세무 판단은 규정/세법 RAG(`tax_refs`)에 위임한다.
- **확장(post-MVP)**: 카드사 제휴로 **MCC** 확보 시 표준 업종코드로 대체·보강한다(§9 Open Issue).

| FastAPI | Agent 오케스트레이션, LLM/ML 호출, 비동기 파이프라인 | 확정 데이터 소유(제안만 반환) |
| FastMCP | Agent용 표준 도구 서버 | 자체 상태 저장 |
| Chroma | 규정/사례 임베딩 저장·유사도 검색 | 트랜잭션 데이터 저장 |
| PostgreSQL | 모든 확정 데이터 SoT | 벡터 검색 |

### 7.2 API 요구사항 개요
**Django DRF (대외, JWT 인증)**: 거래 조회/수집, 증빙 업로드, 정산 조회/보정, 팀 취합·일괄 제출, 정산 최종 확정, 정산 승인/반려/보완, Rule 관리·승인, 역할별 대시보드 지표, ERP 전표(안) 조회.

**FastAPI (내부 전용)**: Draft Agent 실행, Rule Agent 3단계(생성/검증/적용), Risk Review 실행, 모델 학습/추론, 규정문서 임베딩(관리자 트리거). FastAPI는 외부 비공개이며 동기 REST로만 응답해야 한다.

### 7.3 데이터 요구사항 개요
- **PostgreSQL 주요 테이블**: 사용자/팀/역할, 카드, 거래, 증빙, 정산, 정산 상태이력, 규정, Rule/Rule버전, Rule 판정로그, Risk 결과, 학습 라벨, ERP 전표, 감사로그
- **Chroma 컬렉션**: `policy_docs`(사내 규정), `case_history`(과거 사례), `tax_refs`(세법·회계기준)
- 임베딩은 문서 업로드/갱신 시 비동기 처리되어야 하며, Chroma에는 트랜잭션 데이터를 저장하지 않아야 한다.

### 7.4 배포/인프라 요구사항
- React(정적 호스팅) — Django(gunicorn) — FastAPI(uvicorn, FastMCP 포함), Nginx 리버스프록시
- Postgres(주 DB), Chroma(영속 볼륨), Object Storage(증빙 이미지)
- 컨테이너: Docker Compose(MVP) → K8s(확장 시)
- MVP는 Redis/메시지 큐 없이 동기 처리, 외부 의존은 OpenAI API(LLM·비전)

## 9. MVP 제약사항 및 확정된 의사결정

### 9.1 확정 사항
- Django ↔ FastAPI 통신: **동기 REST** (메시지 큐 없음)
- 비동기 스택 **미도입**(Redis/Celery 없음), 무거운 작업은 관리자 온디맨드 배치
- 영수증 처리: 별도 OCR 없이 **OpenAI 비전 모델**로 이미지 직접 판독
- FastMCP: **단일 서버**로 3개 Agent 공용
- Tool 데이터 접근: 관계형=Django 내부 API 경유, 벡터=Chroma 직접
- 예산·정책은 통제(차단) 장치가 아니라 대시보드 지표·추천으로만 반영
- ERP는 MVP 범위에서 **전표(안) 생성까지만** 제공, 실 연동은 제외
- Risk Review는 MVP에서 **단순 이상거래 탐지(비지도, 1차) → RAG 내규 기반 검증(2차)** 구조로 확정. 지도학습(review_probability)·자동 재학습 피드백 루프는 **post-MVP 확장**으로 이관

## 10. 요구사항 매핑 (기존 → 확장, 원본 기획서 §7 인용)

| 영역 | 기존(FR) | 확장 포인트 |
|---|---|---|
| 초안 작성 | (신규) | Draft Agent, 카드구분·분류 자동화, 팀 취합·일괄제출 |
| Rule | FR-RB-*, FR-CL-* | 생성/검증(시뮬레이션)/적용 3단계 명시, 확신 통과·반려 |
| Risk Review | FR-RR-*, FR-RL-* | 미매칭 한정 Review List, **단순 이상탐지(비지도) 1차 → RAG 내규 검증 2차**, 근거 내장 보고서 (지도학습은 post-MVP) |
| 대시보드 | FR-DB-* | 역할별 대시보드 + 정책 결정 추천(통제 아님) |
| 객체 | (암묵) | 사용자·팀·카드구분·분류·규정/예산/증빙 명시적 정의 |

> 본 문서의 §5 FR 번호 체계(FR-DA, FR-RV, FR-RA, FR-ST, FR-UI, FR-ERP, FR-AUD)는 위 기존 코드 체계(FR-RB, FR-CL, FR-RR, FR-RL, FR-DB)를 유지하면서 두 원본 문서에서 코드가 없던 영역을 보완한 것이다.
