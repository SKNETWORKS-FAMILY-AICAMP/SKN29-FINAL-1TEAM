## 0. 문제 정의 (Problem)

법인카드 정산은 **모든 조직이 매달 반복**하지만, 누구도 본업으로 여기지 않는 저부가 업무다. 그럼에도 임직원부터 회계·상부까지 모두의 시간을 잡아먹는다.

### 0.1 역할별로 아픈 지점

```
임직원        영수증 챙기고 · 분류 고르고 · 사유 쓰는 게 일보다 번거롭다
   │           → 월말에 몰아서 입력, 자주 반려 → 재작업
   ▼
팀장          팀원 지출을 취합·검토해 올리는 데 반나절
   │           → 뭐가 문제인지 건건이 열어봐야 한다
   ▼
회계 담당자    수백 건을 규정과 대조하며 전건 수작업 검토
   │           → 대부분 정상인데 다 봐야 하고, 애매한 건에서 시간 소진
   ▼
회계/운영 상부 예산이 어디서 새는지, 규정이 맞는지 데이터가 없다
              → 감(感)으로 정책을 정한다
```

### 0.2 핵심 문제 3가지

| # | 문제 | 결과 |
|---|---|---|
| ① | **입력이 번거롭다** (분류·증빙·사유) | 지연 제출, 반복 반려, 본업 방해 |
| ② | **검토가 전건 수작업이다** | 회계팀 시간의 대부분이 "정상 건 확인"에 소모 |
| ③ | **의사결정 근거가 없다** | 예산·규정 정책을 데이터 없이 감으로 판단 |

> **관통하는 문제**: *정산은 부가가치가 낮은데도 모두의 시간을 잡아먹는다.*
> → 본 서비스는 **① 입력을 AI가 대신하고 ② 확실한 건은 규칙으로 자동 처리하며 ③ 애매한 건만 근거와 함께 골라주고 ④ 결정 근거를 대시보드로 제공**하여 이 문제를 푼다.

## 1. 핵심 객체 모델 (Domain Objects)

### 1.1 사용자 (User) · 역할 (Role)

| 역할 | 설명 | 주 사용 화면 |
|---|---|---|
| **사용자(임직원)** | 실제 카드를 사용하고 지출을 발생시키는 개인 (예: 영업사원, 임원) | 내 지출 / 증빙 제출 |
| **팀(제출 단위)** | 지출을 **취합해서 올리는 단위**. 팀은 다인일 수도, **1인(영업사원·임원 개인)** 일 수도 있다 | 팀 취합 · 제출 |
| **회계 담당자** | 제출된 정산을 **검토·승인·반려**하는 실무자 | 검토 워크스페이스 |
| **회계/운영 상부(관리자)** | 규정·예산 현황을 보고 **정책을 결정**하는 의사결정자 | 거버넌스 대시보드 |

> **핵심 개념 — "팀은 곧 제출 단위"**
> 조직도상 팀이 아니라, **정산을 취합해 올리는 단위**를 팀으로 본다. 영업사원 1인, 임원 1인도 각각 하나의 "팀"으로 취급되어 동일한 취합·제출 플로우를 탄다. → 개인/조직 구분 없이 **하나의 일관된 제출 경험**.

### 1.2 카드 구분 (Card Type)

| 카드 구분 | 정의 | 정산 특성 |
|---|---|---|
| **개인 배정 카드(개인)** | 특정 임직원에게 배정된 법인카드 | 사용자 = 정산 책임자 명확 |
| **팀 카드(팀 공용)** | 팀 단위로 공유하는 카드 | 사용 건별 **실사용자·목적 지정 필요** |
| **공용 카드(공용)** | 부서/법인 공용, 다수가 사용 | 사용자 귀속이 모호 → **AI 사용자 추정·확인** 필요 |
| **후정산 카드/지출(후정산)** | 개인 비용 선지출 후 **사후 청구·정산**(경비 청구) | 영수증 기반, 증빙 의존도 높음 |
| **선결제/충전형(선불·법인 페이 등)** | 사전 충전·한도형 결제 수단 | 잔액·충전 이력 관리 |

> 카드 구분에 따라 **"사용자 귀속"과 "증빙 요구 수준"이 달라진다.** 예) 공용 카드는 실사용자 확인 단계가, 후정산은 증빙 필수 검증이 플로우에 자동 삽입된다.

### 1.3 비용 분류 (Category)

지출은 **업무 성격 기준 분류**로 나뉘며, 분류마다 **규정·예산·증빙 요구**가 다르다.

| 분류 | 예시 | 대표 규정/증빙 포인트 |
|---|---|---|
| **업무활성(업무추진)** | 업무 관련 일반 지출 | 목적·대상 명시 |
| **회의** | 사내/외부 회의, 다과 | 참석자·건당 한도 |
| **식대** | 야근·업무 식사 | 인원수·시간대(야근 등) |
| **출장** | 교통·숙박·일비 | 출장 신청/구간·일정 |
| **접대** | 거래처 접대·경조 | 거래처·접대 한도·세무 리스크 |
| **비품** | 소모품·사무용품 | 자산화 여부·구매 목적 |
| (확장) | 교육/도서, 복리후생 등 | 분류별 규정 확장 가능 |

### 1.4 항목별 3요소: 규정 · 예산 · 증빙

각 **비용 분류(항목)** 는 세 가지 속성을 갖는다. 이 3요소가 Agent들의 판단 재료가 된다.

```
[비용 분류] ──┬─ 규정(Rule 소스)   : 한도, 조건, 필요 증빙, 세무 처리 기준
              ├─ 예산(지표 소스)   : 부서/팀/기간별 배정·소진 (※ 통제 아님, 지표로만)
              └─ 증빙(검증 대상)   : 영수증/명세/사유서 등 필요 서류
```

> **중요**: 예산은 "초과하면 막는" 통제 장치가 **아니다.** 예산은 **대시보드에서 소진 현황·초과 위험을 보여주고, 상부의 정책 판단을 돕는 지표**로만 쓰인다(§5 참조).

### 1.5 정산 건 (Settlement Item)의 라이프사이클(상태)

```
작성중(Draft) → 취합/제출(Submitted) → RPA 1차판정
   ├─ 자동통과(Auto-Pass) ──────► 승인대기(사람 최종확인) → 확정 → ERP 전표(안) 생성*
   ├─ 자동반려(Auto-Reject) ────► 보완요청(Return) → 재제출
   └─ 미매칭/불확실(Needs Review)─► 회계 검토(Review) → 승인/반려/보완
```

> **\* 최종 확정은 사람이 한다**: Rule이 "확신 통과"로 분류해도 자동으로 완료되지 않는다. 회계 담당자가 **최종 확인·확정(올림)** 하는 사람 단계를 반드시 거친다. AI는 검토 부담을 없앨 뿐, 최종 책임은 사람에게 있다.
> **\* ERP 범위(MVP)**: MVP 단계에서는 ERP **직접 적재·연동은 범위에서 제외**하고, 확정된 건에 대해 **ERP 전표(안) 생성**까지만 제공한다(실제 ERP 반영은 향후 과제).

## 2. 3개의 AI Agent (역할 재정의)

플랫폼의 지능은 **역할이 뚜렷한 3개의 Agent**로 구성된다. 각 Agent는 "사람의 시간을 줄인다"는 목적에 봉사한다.

```
┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐
│ ① 초안 작성 Agent │→ │ ② Rule Agent      │→ │ ③ Risk Review     │
│  (Draft Agent)    │   │  (생성/검증/적용) │   │     Agent         │
│  사용자·팀 지원   │   │  회계 규정 자동화 │   │  검토 우선순위    │
└──────────────────┘   └──────────────────┘   └──────────────────┘
   지출 입력 단계          자동 판정 엔진            사람 검토 보조
```

### 2.1 ① 초안 작성 Agent (Draft Agent) — *사용자·팀의 시간을 줄인다*

> **누구를 돕나**: 지출을 올리는 임직원 / 취합해 제출하는 팀(영업사원·임원 1인 팀 포함)
> **무엇을 하나**: 카드 사용 내역을 받아 **정산 초안을 자동 완성**하고, 팀 단위로 **취합·검토 후 한 번에 제출**하도록 돕는다.

**핵심 동작**
- **자동 분류 추천**: 가맹점·금액·시간·패턴을 보고 `식대/회의/출장/접대/비품…` 분류를 자동 제안
- **가맹점 업종 판별**: 가맹점명으로 업종(업태)을 판별해 분류 정확도를 높인다. **자체 DB 캐시 → 카카오 지도 API → 웹검색** 순으로 확정하고 결과를 캐시해 재사용. (업종은 *비용분류 보조 힌트*로만 사용 — 세무 기준 아님. 표준 업종코드 MCC는 카드사 제휴 확장방안)
- **카드 구분 인식**: 개인/팀/공용/후정산에 따라 **필요한 추가 입력(실사용자·목적·증빙)을 자동 요청**
- **증빙 매칭**: 업로드된 영수증을 거래 건과 자동 매칭, 누락 증빙을 **콕 집어 안내**
- **규정 힌트**: 분류별 규정(한도·필요 서류)을 입력 시점에 **미리 알려 반려를 예방**
- **팀 취합 뷰**: 팀원 지출을 한 화면에 모아 **누락·이상 건만 강조**, 팀장이 훑고 **일괄 제출**

**시간 단축 포인트**
> 임직원은 "영수증만 던지면" 초안이 채워지고, 팀은 **문제 있는 건만 손보고 제출** → 나머지는 그대로 통과. **정산에 붙잡히지 않고 메인 업무로 복귀.**

### 2.2 ② Rule Agent — *회계팀의 반복 검토를 없앤다* (생성 / 검증 / 적용)

> **누구를 돕나**: 회계 담당자 (1차 검토 부담 제거)
> **무엇을 하나**: 회계 규정을 **실행 가능한 Rule로 만들고 → 테스트로 검증하고 → 실제 정산에 적용**하여, **확신 가능한 건을 자동 통과/반려**시킨다.

**3단계 책무**

| 단계 | 동작 |
|---|---|
| **생성(Generate)** | 회계 규정·매뉴얼·과거 승인사례를 분석해 단건 룰(노드) 초안 자동 생성 (RAG), 노드를 **룰 그래프(트리)로 조립** |
| **검증(Validate)** | 조립된 **룰 그래프를 과거 거래에 시뮬레이션**하여 오탐/과탐을 테스트, 관리자 검토·승인 후에만 적용 |
| **적용(Apply/RPA)** | 승인된 **ACTIVE 룰 그래프를 순회**해 실거래를 **1차 자동 판정** |

> **룰 도메인 = 그래프(트리)**: 단건 룰은 `조건+액션+next_routings(다음 노드 라우팅)`를 가진 **노드**이고, 이들이 조립된 **룰 그래프**가 "최종 상태 도메인"이다. **ACTIVE 지정·버전관리·롤백은 개별 룰이 아니라 그래프 단위**로 이뤄진다(동시에 ACTIVE인 그래프는 스코프당 하나, 이전 버전으로 즉시 롤백 가능).

**1차 판정 결과(RPA)**
```
확신 통과(Confident Pass)  → 승인대기(상세검토 생략) → 사람 최종확인 후 확정
확신 반려(Confident Reject)→ 자동 보완요청 (사유 자동 안내)
불확실/미매칭             → Risk Review Agent로 이관
```

> **회계팀 효과**: 명백히 규정에 맞는/어긋나는 건은 **사람이 일일이 검토하지 않는다.** 회계팀은 애매한 건에만 개입한다.
> **단, 최종 확정(올림)은 사람의 몫이다.** "확신 통과"는 상세 검토를 생략해 준다는 의미이지 자동 완료가 아니며, 회계 담당자가 **일괄 확인 후 최종 확정**한다. (확정 시 ERP 전표(안) 생성 — ERP 직접 연동은 MVP 제외)

### 2.3 ③ Risk Review Agent — *검토할 건을 골라주고, 근거까지 붙여준다*

> **누구를 돕나**: 회계 담당자 (검토 대상 선별 + 판단 근거 제공)
> **무엇을 하나**: Rule로 걸러지지 않은 **미매칭·불확실 건**에 대해, **검토 우선순위**와 **판단 근거**를 담은 **검토 보고서(Review List)** 를 만든다.

**판단 엔진 (MVP) — ① 단순 이상거래 탐지로 1차 선별 → ② RAG 내규 검증으로 근거 확인**

Risk Review는 두 단계로 동작한다.

| 단계 | 방식 | 역할 |
|---|---|---|
| **① 단순 이상거래 탐지 (1차)** | 비지도 이상탐지로 정상 패턴 대비 이상 스코어링 | *"이 건이 얼마나 평소와 다른가"* 를 정량화해 **이상 후보만 선별** |
| **② RAG 내규 기반 검증 (2차)** | 이상 후보에 대해 사내 규정·세법·유사 사례 검색·대조 | *"실제로 내규 위반인가, 왜 봐야 하는가"* 를 검증·설명 |

**① 단순 이상거래 탐지 (Anomaly Detection, 비지도)**
- 규정에 명시되지 않은 **정상 패턴에서 벗어난 이상 거래**를 통계·학습 기반으로 포착 (Rule은 "아는 규칙"만, 이상탐지는 "규칙화되지 않은 이상"까지)
- **학습 신호(MVP)**: **라벨 없이도** 금액·빈도·시간·거래처·사용자 패턴의 **일탈(outlier)** 자체를 스코어링 (예: Isolation Forest 계열). 라벨 의존적 지도학습은 사용하지 않는다.
- **탐지하는 이상 유형(예)**: 한도 임계값 바로 아래 반복 분할, 심야·휴일 비정상 사용, 동일 거래처 급증, 사용자 평소 패턴 대비 급변, 중복 결제 의심
- **출력**: 거래별 **이상 스코어 → 컷오프 이상 건만 "이상 후보"로 선별**, 이상 사유(어느 피처가 튀었는지)를 함께 제시

**② RAG 내규 기반 검증 (이상 후보 한정)**
- 1차에서 걸러진 이상 후보에 대해서만 사내 규정·세법·유사 사례를 검색하고, LLM이 거래를 내규와 대조해 **위반 여부를 검증**한 뒤 검토사유·권장의견을 근거(출처 조항)와 함께 생성

> **Rule과의 차이**: Rule Agent는 *알려진 규칙*으로 확신 건을 처리하고, 이상탐지는 *규칙으로 못 잡는 미지의 이상*을 찾아 1차 선별하며, RAG 검증이 그 이상 후보의 내규 근거를 붙인다. 셋은 상호 보완적이다.

> **확장(post-MVP)**: 회계 담당자의 승인/반려 결과(`decision_labels`)가 쌓이면, 이를 학습한 **지도학습 "검토 필요 확률" 모델**과 자동 재학습 피드백 루프를 추가해 우선순위 산출을 고도화한다. MVP는 라벨을 **축적만** 하고 재학습에는 쓰지 않는다.

**보고서 1건에 담기는 것**
- **이상 스코어** · 우선순위 · 내규 검증 결과
- 검토가 필요한 **사유**(무엇이 걸리는지 — 어떤 피처가 이상한지 포함)
- 관련 **사내 규정·세법** 근거 (RAG 자동 연결)
- **유사 과거 승인/반려 사례**
- **AI 권장 의견** (승인/보완/반려 제안)

> **회계팀 효과**: 전건을 뒤지지 않고 **위험순으로 정렬된 목록**을 위에서부터 처리. 각 건의 근거가 붙어 있어 **문서 찾는 시간이 사라진다.**

## 5. 대시보드 고도화 — 역할별 지표 · 인사이트 · 정책 추천

> **원칙**: 예산·정책은 **막지 않는다.** 대시보드가 **현황과 위험을 보여주고, 상부가 정책을 결정하도록 근거·추천**을 제공한다.

### 5.1 역할별 대시보드 (관심사가 다르다)

| 역할 | 핵심 관심 | 대표 지표 |
|---|---|---|
| **임직원** | 내 처리 상태 | 미제출·보완요청·평균 승인 소요 |
| **팀장** | 팀 취합 효율 | 팀 이상건율·제출 리드타임·반복 반려 사유 |
| **회계 담당자** | 검토 부하 | 자동처리율·검토 대기·평균 검토 시간 |
| **회계/운영 상부** | **정책 결정** | 분류별 지출 추세·예산 소진·정책 위반 패턴·리스크 |

### 5.2 상부용 거버넌스 대시보드 (정책 결정 지원)

```
┌─ 비용 거버넌스 (2026 Q3) ────────────────────────────────┐
│ 자동처리율 82% ▲   회계 검토시간 -41% ▲   반복반려 -28% ▲ │
├───────────────────────────┬──────────────────────────────┤
│ 분류별 지출·예산 소진      │ 정책 인사이트 (추천)          │
│ 접대  ███████░ 예산 96% ⚠ │ 💡 접대비, 3개팀에서 소진 임박 │
│ 출장  █████░░░ 예산 61%    │   → 한도 재검토 or 예산 재배분 │
│ 식대  ████░░░░ 예산 48%    │ 💡 '야근 식대' 반복 승인 320건 │
│ 비품  ██████░░ 예산 73%    │   → 자동승인 Rule 신설 추천    │
│                           │ 💡 접대 한도-1만원 밑 분할결제  │
│                           │   12건 → 한도 회피 패턴 점검   │
└───────────────────────────┴──────────────────────────────┘
```

**대시보드가 제공하는 3종 인사이트**
1. **예산 지표(모니터링)**: 분류/팀/기간별 소진율·초과 위험 — *경고만, 차단 없음*
2. **정책 인사이트(추천)**: "이 패턴은 Rule로 만들 만하다 / 이 한도는 재검토가 필요하다"
3. **리스크 인사이트(탐지)**: 한도 회피성 분할, 특정 분류 급증 등 **상부가 봐야 할 신호**

> 상부는 이 화면을 근거로 **규정 개정 · 예산 재배분 · 신규 Rule 도입**을 판단한다. 시스템은 **추천**하고, **결정은 사람이** 한다.

### 5.3 운영 KPI (시간 절감 중심)

- 자동 처리율 / 회계 검토 건수·시간 감소율
- 정산 제출→승인 평균 리드타임
- 반복 반려 감소율 / 증빙 누락 사전 예방 건수
- 신규 Rule 발굴·적용 건수 및 절감 효과

## 6. 카드 구분 × 분류에 따른 플로우 분기 (요약)

| 상황 | 플로우 자동 분기 |
|---|---|
| **공용 카드** | Draft Agent가 **실사용자·목적 지정** 단계 삽입 |
| **후정산** | **증빙 필수 검증** 강화, 영수증 없으면 제출 차단(안내) |
| **접대 분류** | 한도·거래처·세무 리스크 → Risk Review 우선순위 상향 |
| **출장 분류** | 출장 신청/일정과 대사, 증빙(교통·숙박) 매칭 |
| **소액·정형 식대** | Rule Agent 확신 통과 → 상세검토 생략, 사람 최종확인 후 확정 |

## 7. 요구사항 매핑 (기존 → 확장)

| 영역 | 기존(FR) | 확장 포인트 |
|---|---|---|
| 초안 작성 | (신규) | Draft Agent, 카드구분·분류 자동화, 팀 취합·일괄제출 |
| Rule | FR-RB-*, FR-CL-* | 생성/**검증(시뮬레이션)**/적용 3단계 명시, 확신 통과·반려 |
| Risk Review | FR-RR-*, FR-RL-* | 미매칭 한정 Review List, **단순 이상탐지(비지도) 1차 → RAG 내규 검증 2차**, 근거 내장 보고서 (지도학습은 post-MVP) |
| 대시보드 | FR-DB-* | **역할별 대시보드 + 정책 결정 추천**(통제 아님) |
| 객체 | (암묵) | 사용자·팀·카드구분·분류·규정/예산/증빙 **명시적 정의** |

## 8. 향후 확정 필요 사항 (Open Issues)

1. **카드 구분 표준화**: 개인/팀/공용/후정산/선불의 실제 운영 구분과 필드 정의
2. **"팀 = 제출 단위" 모델**: 조직도와의 매핑, 1인 팀 처리 규칙
3. **분류 체계 확정**: 업무활성·회의·식대·출장·접대·비품 외 확장 분류와 분류별 규정 서식
4. **Rule 확신 임계치**: 자동 통과/반려로 볼 신뢰도 기준
5. **증빙 필수도**: 카드구분·분류별 증빙 요구 수준(후정산 vs 소액 식대 등)
6. **대시보드 추천의 권한 경계**: AI 추천을 상부가 원클릭 적용할지, 결재를 거칠지
7. **(MVP) 단순 이상탐지 모델 선정**: 비지도 이상탐지 알고리즘·피처셋·이상 스코어 컷오프 확정
8. **(post-MVP) 지도학습 도입**: 검토필요확률 모델 도입 시점·필요 라벨량·자동 재학습 주기 확정
9. **ERP 연동 범위**: MVP는 **ERP 전표(안) 생성까지**로 한정, 실제 ERP 적재·연동 방식은 향후 확정
10. **가맹점 업종 캐시 정책**: 카카오 API 키·쿼터, 캐시 TTL·정규화 규칙, 업종→분류 매핑 테이블
11. **(확장) 카드사 제휴 MCC 확보**: 세무·회계 기준 표준 업종코드 도입(카카오/웹 업종을 대체·보강)
12. **룰 그래프 정합성**: next_routings 사이클 방지·도달불가 노드 검출, 다중 라우팅 우선순위, 그래프 스코프(전사/분류/본부) 정의
