## 1. 아키텍처 개요

### 1.1 설계 원칙

- **관심사 분리**: **트랜잭션/도메인(Django)** 과 **AI 오케스트레이션(FastAPI)** 을 물리적으로 분리한다.
- **System of Record는 Postgres 하나**: 모든 확정 데이터의 단일 진실 원천(SoT)은 Django가 관리하는 PostgreSQL이다. AI는 **제안(초안·스코어·근거)** 만 생성하고, 확정은 Django 도메인을 통해서만 이뤄진다.
- **Agent = LLM + Tools(MCP)**: 모든 Agent는 **단일 FastMCP 서버**로 노출된 표준 도구를 통해서만 데이터에 접근한다. (LLM의 직접 DB 접근 금지 → 감사·권한 일원화)
- **MVP는 동기 REST**: 별도 메시지 큐(Redis 등) 없이 **동기 REST 호출**로 처리한다. 문서 임베딩·모델 학습 등 무거운 작업은 **관리자 트리거 배치(온디맨드)** 로 단순화한다.
- **사람 확정 원칙**: 자동 통과 건도 최종 확정은 사람이 수행(기획서 §1.5). 시스템은 상태만 `승인대기`로 전이시킨다.

### 1.2 시스템 구성도

```
                    ┌─────────────────────────────────────────────┐
                    │              React SPA (Frontend)            │

                    │  · System of Record (PostgreSQL Owner)       │
                    └───────┬───────────────────────────┬─────────┘
                            │ 내부 REST (AI 작업 위임)    │ ORM
                            │                            │
            ┌───────────────▼──────────────┐    ┌────────▼─────────┐
            │   FastAPI (AI Orchestrator)   │    │   PostgreSQL     │

        │  · Rule 실행     │        ┌──────────────┐
```

### 1.3 컴포넌트 책임 분담 (RACI 요약)

| 컴포넌트 | 책임 | 하지 않는 것 |
|---|---|---|
| **React** | 화면, 상태표시, 사용자 액션 | 비즈니스 규칙 판단 |
| **Django(DRF)** | 인증·RBAC, 도메인 CRUD, 정산 상태머신, ERP 전표(안), Postgres 소유 | LLM 직접 호출·ML 추론 |
| **FastAPI** | Agent 오케스트레이션, LLM/ML 호출, 비동기 파이프라인 | 확정 데이터 소유(제안만 반환) |
| **FastMCP** | Agent용 표준 도구 서버(검색·조회·Rule실행·OCR) | 자체 상태 저장(무상태 지향) |
| **Chroma** | 규정/사례 임베딩 저장·유사도 검색(RAG) | 트랜잭션 데이터 저장 |
| **PostgreSQL** | 모든 확정 데이터 SoT | 벡터 검색 |

> **왜 Django + FastAPI 둘 다?**
> Django는 인증·ORM·Admin·상태머신 등 **정형 도메인**에 강하고, FastAPI는 async·LLM/ML 서빙 등 **AI 워크로드**에 강하다. Django를 SoR로 고정하고 AI 작업만 FastAPI로 위임하면 **책임 경계와 감사 지점이 명확**해진다.

## 2. 기술 스택 상세

| 계층 | 기술 | 용도 | 선정 이유 |
|---|---|---|---|
| Frontend | **React** (+ TS, 상태관리, 차트 lib) | SPA UI, 역할별 대시보드 | 컴포넌트 재사용, 대시보드 시각화 |
| Core API | **Django 5 + DRF** | 도메인/인증/상태머신/Admin | ORM·마이그레이션·RBAC·Admin 성숙 |
| AI API | **FastAPI** | Agent 오케스트레이션·ML 서빙 | async, pydantic, LLM 생태계 |
| Agent Tools | **FastMCP (단일 서버)** | 3개 Agent 공용 도구 노출 | MCP 표준화, tool 재사용/감사 |
| LLM/Vision | **OpenAI (GPT 비전 모델)** | 분류·근거 생성 + **영수증 이미지 직접 판독** | 별도 OCR 엔진 불필요(비전으로 대체) |
| Vector DB | **Chroma** | 규정/FAQ/사례 임베딩·RAG 검색 | 경량, 빠른 프로토타이핑 |
| RDB | **PostgreSQL** | 트랜잭션 SoT, 이력/라벨, **가맹점 업종 캐시** | JSONB·인덱싱·신뢰성 |
| 외부 API | **카카오 로컬(지도) API · Web Search** | 가맹점 업종(업태) 판별 → 비용분류 보조 | 자체 규정 외 가맹점 메타데이터 확보(캐시화) |

> **MVP 단순화**: 별도 OCR 엔진·메시지 큐(Redis/Celery)는 두지 않는다. 영수증은 **OpenAI 비전 모델이 이미지에서 바로 필드 추출**하고, 처리 흐름은 **동기 REST**로 통일한다.

## 3. 데이터 아키텍처

### 3.1 PostgreSQL — 주요 테이블(개요)

| 도메인 | 테이블(예) | 핵심 컬럼 |
|---|---|---|
| 조직/사용자 | `users`, `teams`, `team_members`, `roles` | role, team_id(제출단위) |
| 카드 | `cards` | card_type(개인/팀/공용/후정산/선불), owner_id |
| 거래 | `transactions` | merchant, amount, ts, card_id, raw_payload(JSONB) |
| 가맹점 업종 캐시 | `merchant_categories` | normalized_name(idx), place_id, industry_code, industry_label, source(CACHE/KAKAO/WEB), confidence, resolved_at, raw(JSONB) |
| 증빙 | `receipts` | file_ref, ocr_text, matched_tx_id, status |
| 정산 | `settlements` | tx_id, category, status(상태머신), submitted_by |
| 정산 상태이력 | `settlement_events` | from_state, to_state, actor, reason, ts |
| 규정 | `policies`, `policy_docs` | category, limit, required_evidence |
| Rule | `rules`, `rule_versions` | condition(JSON/DSL), action, status, sim_result |
| Rule 판정로그 | `rule_hits` | tx_id, rule_id, decision, confidence |
| Risk 결과 | `risk_reviews` | tx_id, anomaly_score, reasons(JSONB) *(review_prob 컬럼은 post-MVP 지도학습 도입 시)* |
| 학습 라벨 | `decision_labels` | tx_id, label(승인/반려/보완/수정), actor, ts |
| ERP | `erp_vouchers` | settlement_id, voucher_payload(JSONB), status(안/확정) |
| 감사 | `audit_logs` | actor, action, target, before/after, ts |

### 3.2 Chroma — 벡터 컬렉션(개요)

| 컬렉션 | 소스 | 용도 | 메타데이터 |
|---|---|---|---|
| `policy_docs` | 사내 회계 규정·매뉴얼·FAQ | Rule 생성 / Risk 근거 RAG | category, version, effective_date |
| `case_history` | 과거 승인·반려 사례 | 유사 사례 검색 | decision, category, amount_band |
| `tax_refs` | 세법·회계 기준 | 세무 근거 RAG | source, article |

> 임베딩은 **문서 업로드/갱신 시 비동기로 청킹→임베딩→upsert**. Chroma에는 원문 조각 + 메타데이터만, 트랜잭션 데이터는 절대 넣지 않는다.

### 3.3 정산 상태 머신 (Django 관리)

```
DRAFT → SUBMITTED → RPA_JUDGED
  ├─(auto-pass)──► PENDING_CONFIRM ─(사람 확정)─► CONFIRMED → ERP_VOUCHER_DRAFTED
  ├─(auto-reject)► RETURNED ─(재제출)─► SUBMITTED
  └─(needs-review)► IN_REVIEW ─► APPROVED / RETURNED / CORRECTED
```

- 상태 전이는 **Django 서비스 레이어에서만** 수행, 모든 전이는 `settlement_events` + `audit_logs`에 기록.

## 4. Agent 아키텍처 & Flow

### 4.0 공통: Agent 실행 모델

```
Django(도메인 이벤트) ──REST──► FastAPI(Agent Orchestrator)
                                    │
                                    ├─ LLM 호출 (프롬프트 + tool schema)
                                    ├─ FastMCP tool call (검색/조회/Rule/OCR)
                                    ├─ ML Serving (이상탐지 추론)
                                    └─ 결과(제안) ──REST──► Django (저장/상태전이)
```

- **모든 Agent 출력은 "제안"** 이며 Postgres에 결과 테이블로 적재되지만 **확정 상태로 만들지 않는다.**
- MVP는 **동기 호출**: Django가 FastAPI를 REST로 호출하고 응답을 그대로 받아 저장한다. (별도 Job 큐 없음. LLM/도구 호출 trace는 로그로 남겨 감사·비용추적.)

---

### 4.1 ① 초안 작성 Agent (Draft Agent)

**목적**: 거래·영수증을 받아 정산 초안(분류·증빙매칭·규정힌트)을 자동 완성.

**Flow**
```
1. 거래 수집 (카드사 원천/업로드) ──► transactions 적재 (Django)
2. 영수증 이미지 업로드 ──► settlements 첨부 (Django)
3. Draft Agent 실행 (FastAPI)
     ├─ OpenAI 비전: 영수증 이미지 → 필드 추출(가맹점/금액/일시/품목) + 거래 매칭
     ├─ FastMCP:classify_merchant(가맹점)        업종 판별(캐시→카카오 지도→웹검색, §7-1)
     ├─ LLM: 분류 추론 (merchant/업종/amount/time/pattern → category)
     ├─ FastMCP:get_policy(category)             분류별 규정·필요증빙 조회
     ├─ FastMCP:get_card_context(card_id)        카드구분별 필요입력 판정
     └─ 산출: {category, merchant_industry, matched_receipt, missing_evidence[], policy_hints[]}
4. 결과 ──► settlements(DRAFT) 초안 저장 (Django)
5. React: 임직원 화면에 초안 표시 → 사용자 보정 → 팀 취합 → SUBMITTED
```

**입력**: transaction, receipt 이미지, card_type, 사용자/팀 컨텍스트
**출력**: category(추천), merchant_industry(업종), 증빙 판독·매칭결과, 누락 증빙 목록, 규정 힌트
**사용**: OpenAI 비전(영수증 판독·매칭), FastMCP `classify_merchant`·`get_policy`·`get_card_context`
**분기**: 카드구분 공용→실사용자 지정 요청 / 후정산→증빙 필수 플래그

---

### 4.2 ② Rule Agent (생성 / 검증 / 적용)

**목적**: 규정을 실행가능 Rule로 만들고(생성), 과거 데이터로 검증하고(검증), 실거래를 1차 자동 판정(적용).

#### (a) 생성 Flow
```
관리자: 규정문서 업로드 ─► 비동기 임베딩 ─► Chroma(policy_docs)
Rule Agent(생성):
  ├─ FastMCP:search_policy(query)  RAG로 규칙화 가능 조항 추출
  ├─ LLM: Rule 초안 생성 (condition DSL + action)
  └─ rules(status=DRAFT) 저장
```

#### (b) 검증 Flow (적용 전 필수)
```
Rule Agent(검증):
  ├─ FastMCP:fetch_historical_tx(기간, 필터)   과거 거래 로드
  ├─ Rule Engine: 후보 Rule을 과거 거래에 시뮬레이션
  ├─ 지표 산출: 매칭건수 / 오탐율 / 예상 검토감소량
  └─ rule_versions.sim_result 저장 ─► 관리자 검토·승인 화면(4.5 콘솔)
관리자 승인 시에만 rules(status=ACTIVE) 전이
```

#### (c) 적용 Flow (RPA 1차 판정)
```
SUBMITTED 정산 ─► Rule Engine 평가
  ├─ 확신 통과(conf ≥ θ_pass)  ─► PENDING_CONFIRM (상세검토 생략, 사람 최종확정 대기)
  ├─ 확신 반려(conf ≥ θ_reject)─► RETURNED (보완요청 자동 안내)
  └─ 미매칭/저신뢰            ─► Risk Review Agent로 이관
rule_hits 로그 기록
```

**입력**: 규정문서(생성), 과거거래(검증), 제출정산(적용)
**출력**: Rule 초안, 시뮬레이션 지표, 거래별 판정+confidence
**사용 Tool**: `search_policy`, `fetch_historical_tx`, `run_rule_engine`
**핵심 파라미터**: `θ_pass`, `θ_reject`(확신 임계치 — Open Issue)
**중요**: Rule은 결정론적 엔진 실행이 원칙, LLM은 **생성 단계에서만** 사용(적용 단계는 재현성 확보).

### 4.3 ③ Risk Review Agent (단순 이상탐지 1차 + RAG 내규 검증 2차)

**목적**: Rule로 안 걸러진 미매칭·불확실 건에 검토 우선순위와 근거를 부여.

**Flow (MVP: 2단계)**
```
미매칭 정산 ─► Risk Review Agent(FastAPI)
  [1차: 단순 이상거래 탐지]
  1. Feature 조립
       └─ FastMCP:get_tx_features(tx)  금액/시간/요일/가맹점/사용자패턴/이력
  2. 비지도 이상탐지 추론 (ML Serving)
       └─ 단순 이상탐지 모델(예: Isolation Forest 등): anomaly_score + 피처기여
       └─ 컷오프 이상 건만 "이상 후보"로 1차 선별
  [2차: RAG 내규 기반 검증 — 이상 후보 한정]
  3. 근거 수집 (RAG)
       ├─ FastMCP:search_policy(category, features)   관련 규정·세법
       └─ FastMCP:search_cases(tx)                    유사 승인·반려 사례
  4. LLM: 내규와 대조해 위반 여부 검증 → 검토사유 + 권장의견(승인/보완/반려) 생성
  5. risk_reviews 저장 ─► 회계 검토 워크스페이스(우선순위순 정렬)
회계 담당자 결정 ─► decision_labels 적재(향후 지도학습용 데이터로 축적만)
```

**입력**: 미매칭 정산 + 파생 feature
**출력**: anomaly_score, 이상사유(피처기여), 규정·사례 근거(출처 조항), AI 권장의견
**사용 Tool**: `get_tx_features`, `ml_infer`, `search_policy`, `search_cases`
**모델 운영(MVP)**: 라벨 불필요한 비지도 이상탐지 온라인 추론(실시간). 컷오프·모델 갱신은 관리자 온디맨드.

> **확장(post-MVP)**: 라벨이 충분히 쌓이면 지도학습 모델(검토필요확률 `review_probability`, 반려/보완 라벨 학습)과 `decision_labels` 기반 자동 재학습 피드백 루프를 추가한다. MVP는 지도학습·재학습 루프를 포함하지 않는다.

## 5. FastMCP 도구(Tool) 명세

> **단일 FastMCP 서버**가 3개 Agent의 도구를 모두 노출한다. (Agent별 분리 안 함 — MVP 단순화)
> 영수증 판독은 별도 tool이 아니라 **OpenAI 비전 모델을 Draft Agent가 직접 호출**한다.

| Tool | 인자(LLM이 넘김) | 출력 | 데이터 원천 | 사용 Agent |
|---|---|---|---|---|
| `get_policy` | category | limit, required_evidence, refs | Postgres(via Django) | Draft, Rule, Risk |
| `get_card_context` | card_id | card_type, required_inputs | Postgres(via Django) | Draft |
| `classify_merchant` | merchant, place_hint? | industry_code, industry_label, confidence, source | 캐시=Postgres(via Django), 미스 시 카카오 지도/웹검색 | Draft, Risk |
| `search_policy` | query, filters | chunks[] | **Chroma 직접** | Rule, Risk |
| `search_cases` | tx / query | similar_cases[] | **Chroma 직접** | Risk |
| `fetch_historical_tx` | period, filters | tx[] | Postgres(via Django) | Rule(검증) |
| `run_rule_engine` | tx, ruleset | decision, confidence, hits[] | 내부 Rule 엔진 | Rule(적용) |
| `get_tx_features` | tx_id | feature_vector | Postgres(via Django) | Risk |
| `ml_infer` | feature_vector | anomaly_score, contribs *(MVP: 비지도 이상탐지만. review_prob는 post-MVP)* | ML 모델(FastAPI 내) | Risk |

### 5.1 Tool의 데이터 접근 경로 (중요)

Tool은 **무상태 함수처럼 보이지만, LLM이 넘기는 인자만으로는 부족**하다. 예) `get_tx_features(tx_id)` 는 인자로 `tx_id` 하나만 받지만, 실제로는 그 거래의 상세·**사용자의 최근 이력·동일 거래처 누적액**까지 조회해야 feature가 만들어진다. 이때 **"인자 외 원본 데이터를 어디서 가져오느냐"** 가 데이터 접근 경로다.

**원칙: 데이터 종류에 따라 접근 경로를 이원화한다.**

```
┌──────────────── FastMCP Tool 구현체 ────────────────┐
│                                                     │
│  관계형 데이터(거래·이력·규정·정산·카드)             │
│     └─► Django 내부 조회 API (read-only REST) ──► Postgres
│         · Postgres를 만지는 것은 Django 하나 (SoR 유지)
│         · ORM·권한·스키마를 Django에 일원화
│                                                     │
│  벡터 데이터(규정/사례 임베딩)                       │
│     └─► Chroma 클라이언트 직접 ──► Chroma            │
│         · Chroma는 Django가 소유하지 않으므로 직접 접근
│                                                     │
└─────────────────────────────────────────────────────┘
```

| 데이터 | 접근 경로 | 이유 |
|---|---|---|
| 관계형(Postgres) | **Django 내부 read API 경유** | Postgres 접근점을 Django로 단일화(SoR·권한·감사) |
| 벡터(Chroma) | **Chroma 직접** | Django가 소유하지 않는 저장소, 홉 최소화 |
| ML 추론 | FastAPI 프로세스 내부 호출 | 별도 서비스 불필요(MVP) |

> **즉, LLM/Tool은 절대 Postgres에 직접 SQL을 쏘지 않는다.** 관계형 데이터가 필요하면 반드시 **Django 내부 API**를 거친다. Chroma·ML만 tool이 직접 접근한다.

**설계 규칙**
- Tool은 **무상태**, 인자·반환은 pydantic 스키마로 엄격 정의.
- 관계형 데이터 접근은 **Django 내부 API 경유**(LLM에 raw DB 노출 금지).
- 모든 tool call은 **로그로 기록**(감사·비용·디버깅).

## 7. 위험 검토 파이프라인 (MVP: 이상탐지 1차 + RAG 내규 검증 2차)

MVP는 라벨이 충분히 축적되기 전(콜드스타트)이므로 **지도학습에 의존하지 않는 2단계** 구조로 단순화한다.

```
[1차 · 단순 이상탐지 (비지도, 라벨 불필요)]
  [학습]  transactions ──► Feature 생성 ──► 비지도 이상탐지 학습(예: Isolation Forest 등)
         ──► 모델 레지스트리(버전관리) ──► ML Serving 배포
  [추론]  실시간(동기): get_tx_features → ml_infer → anomaly_score/피처기여
         ──► 컷오프 이상 건만 "이상 후보" 선별
[2차 · RAG 내규 검증 (이상 후보 한정)]
  이상 후보 ──► search_policy/search_cases(RAG) ──► LLM 내규 대조·검증
         ──► 검토사유·권장의견(출처 포함) ──► risk_reviews 저장
[피드백] 회계 결정 → decision_labels 적재만 (MVP는 재학습 안 함, 향후 지도학습용)
```

**Feature 예시**: 금액·정규화금액, 시간대/요일/심야여부, 가맹점/업종, 사용자·부서, 최근 N일 빈도, 동일거래처 누적, Rule 적용결과, 증빙 유무.
**평가지표**: 이상탐지 검토 적중률·AUC, RAG 근거 정확도, 검토량 감소율. (목표치 Open Issue)

> **확장(post-MVP)**: 라벨이 쌓이면 지도 검토필요확률 분류기(XGBoost/LogReg 등)와 `decision_labels` 기반 온디맨드 재학습 루프를 추가한다.

## 7-1. 가맹점 업종 구분 파이프라인 (Merchant Classification)

가맹점명만으로는 비용분류가 모호한 경우가 많아, **가맹점의 업종(업태)** 을 별도로 판별해 Draft Agent의 비용분류 제안과 Risk 이상탐지 feature('가맹점/업종')를 보강한다. 외부 조회 비용·지연을 줄이기 위해 **자체 DB 캐시를 1차 소스**로 두는 캐스케이드 구조다.

```
가맹점명(+장소 힌트)
  0. 정규화        지점명·특수문자·체인 접미 제거 → 조회 키 생성
  1. [캐시]         merchant_categories(자체 DB) 히트 & 유효(TTL) → 반환        source=CACHE
  2. [카카오 지도]   로컬 키워드검색 → category_group_code/category_name 매핑 → 반환  source=KAKAO
  3. [Web Search]   1·2 실패/모호 시 웹검색 → LLM이 스니펫 해석 → 업종 추론        source=WEB
  4. [확정+캐시]     업종·신뢰도·source·조회일 upsert → 이후 동일 가맹점 재사용
```

- **출력**: `{ industry_code, industry_label, confidence, source }`
- **업종→분류 매핑 예**: FD6 음식점→식대/회의 · CE7 카페→회의/식대 · AD5 숙박→출장 · 주류·유흥→접대(리스크↑) · 문구·전자→비품
- **활용**: ① Draft Agent 비용분류 제안(가맹점+업종 → 분류) ② Risk 이상탐지 feature 보강(업종-분류 불일치, 유흥업종 접대 등)
- **캐시(자체 DB)**: 관계형이므로 Django 소유(`merchant_categories`). 외부 API(카카오/웹검색) 호출은 **FastAPI(ai)** 에서 수행하고, 결과만 Django 경유로 캐시 적재한다(§5.1 원칙 준수).
- **⚠ 세무/회계 부적합**: 카카오·웹 기반 업종은 **장소·마케팅 분류**라 세무 계정과목과 불일치할 수 있다. **비용분류 보조 힌트로만** 사용하고, 세무 판단은 규정/세법 RAG(`tax_refs`)에 위임한다.
- **확장(post-MVP)**: 카드사 제휴로 **MCC(Merchant Category Code)** 확보 시 표준 업종코드로 대체·보강한다(§11.2).

## 8. RAG 파이프라인

```
문서 업로드 ─► 청킹 ─► 임베딩 ─► Chroma upsert (메타데이터 포함)
질의 시    ─► 쿼리 임베딩 ─► Chroma 유사도 검색(top-k, 메타 필터) ─► LLM 컨텍스트 주입
```

- **버전/유효일 메타**로 최신 규정 우선. 규정 개정 시 재임베딩(비동기).
- 근거 표시 시 **출처(문서·조항) 함께 반환**하여 회계 담당자가 검증 가능하게.

### 11.1 MVP 확정 사항 (이번 결정)

- ✅ **Django ↔ FastAPI 통신**: **동기 REST** (메시지 큐 없음)
- ✅ **비동기 스택**: **미도입** (Redis/Celery 없이 동기 처리, 무거운 작업은 관리자 온디맨드 배치)
- ✅ **영수증 처리**: 별도 OCR 없이 **OpenAI 비전 모델**로 이미지 직접 판독
- ✅ **FastMCP**: **단일 서버**로 3 Agent 공용
- ✅ **Tool 데이터 접근**: 관계형=**Django 내부 API 경유**, 벡터=**Chroma 직접** (§5.1)
- ✅ **가맹점 업종 구분**: **자체 DB 캐시 → 카카오 지도 API → 웹검색** 캐스케이드로 판별, 비용분류 **보조 힌트**로만 사용(세무 판단 아님). MCC는 post-MVP (§7-1)

### 11.2 남은 확정 필요 사항

1. **확신 임계치 θ_pass/θ_reject** 및 이상탐지 스코어 컷오프
2. **(MVP 1차) 비지도 이상탐지 알고리즘·피처셋·컷오프** 최종 선정 및 목표 성능치
2-1. **(post-MVP) 지도학습(review_probability) 도입 시점·필요 라벨량·자동 재학습 주기**
3. **임베딩 모델**(OpenAI vs 로컬) 및 Chroma 영속·백업 전략
4. **ERP 전표(안) 스키마**: MVP 산출물 포맷(연동은 범위 외)
5. **멀티테넌시**: 테넌트별 데이터·모델·규정 격리 범위(MVP 필요 여부 포함)
6. **OpenAI 모델 선정**(비전·LLM) 및 프롬프트·비용 관리 방식
7. **가맹점 업종 캐시 정책**: 카카오 API 키·쿼터, 캐시 TTL·정규화 규칙, 업종→분류 매핑 테이블 확정
8. **(확장) 카드사 제휴 MCC 확보**: 세무·회계 기준 업종코드(MCC) 도입 방식·시점 — 카카오/웹 업종을 대체·보강
